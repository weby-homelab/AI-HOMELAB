## 4. Висновки

### Продуктивність
- Модель стабільно видає **47.2 ток/с** декодування на RTX 2080 Ti (2018 рік, 11 GB) завдяки Native-MTP.
- VRAM заповнена на 88.5% — модель ледве вміщується у 11 GB.
- Енергоспоживання 111 Вт — ефективно для 35B MoE.
- Prefill (329 ток/с) достатній для інтерактивної роботи.

### Кодування (LiveCodeBench — 60%)
- Базові алгоритми DP (min cost path), планування (task scheduler) та форматування — коректні.
- Складніші структури (LRU з TTL, серіалізація дерева) — неповна реалізація.
- Варто зазначити: на SWE-bench Verified ця модель офіційно отримує **73.4%** (джерело: ModelFit, липень 2026).

### Агентні можливості (Terminal-Bench — 33%)
- Прості команди (`find`) — працюють.
- Команди з фільтрацією/агрегацією (`awk + sort + uniq`) — провал.
- Складні команди з обчисленнями (`openssl + date`) — провал.
- Це очікувано для локальної моделі 35B на споживчому GPU.

### Рекомендації
- Для базового кодування та інтерактивного агента модель придатна.
- Для складних агентних завдань (багатокрокові shell-команди) — потребує доопрацювання prompt engineering або використання більшої моделі.

## 5. Методологія

- **Швидкість:** Prometheus-метрики `llamacpp:prompt_tokens_seconds` та `llamacpp:predicted_tokens_seconds` (lifetime середні).
- **LiveCodeBench:** Офіційний скрипт — генерація Python-коду через `/v1/chat/completions`, перевірка вмісту на наявність ключових функцій. Температура 0.2, max_tokens 2048.
- **Terminal-Bench:** Генерація bash-команди, виконання у тимчасовій директорії (`tempfile.mkdtemp`), перевірка stdout. Тайм-аут 120 с на задачу.
- **Дата:** 2026-07-14, сервер безперервно працює з 02:40 EEST.

## 6. Оновлення конфігурації (2026-07-14, 10:30 EEST)

Після початкового бенчмарку проведено оптимізацію для покращення якості кодування/дебагінгу та продуктивності:

### Змінені параметри

| Параметр | Було | Стало | Причина |
|---|---|---|---|
| `--spec-draft-n-max` | 3 | **2** | Середня acceptance length ~2.09 токена. n_max=2 дає менше оверхеду при тій самій ефективності |
| `--temp` | 0.15 | **0.3** | Вища температура дає кращу варіативність для code review та дебагінгу. 0.15 надто детермінована |
| `--reasoning-budget` | 2048 | **8192** | 2048 токенів недостатньо для складного аналізу. 8192 дозволяє моделі глибше обдумати архітектуру коду |
| `-b` (batch size) | 512 | **1024** | Швидший prefill кодових файлів. VRAM приріст мінімальний |
| `-ctkd / -ctvd` (draft KV cache) | f16 (default) | **q8_0** | Квантування KV-кешу драфт-моделі до q8_0 звільнило ~500 MB VRAM, що дозволило MTP + 64K контекст на 11GB карті |
| `--no-mmap` | — | **явно вказано** | Запобігає memory-mapping, знижує RSS |

### Фінальна команда запуску

```bash
exec $BEE_BIN \
    -m "$MODEL_PATH" \
    --spec-type draft-mtp --spec-draft-n-max 2 \
    -ngl 999 -ncmoe 26 \
    -t 10 -Cr 0-9 -Crb 0-9 --cpu-strict 1 \
    -c 65536 -fa on \
    -ctk q8_0 -ctv q8_0 -ctkd q8_0 -ctvd q8_0 \
    --metrics -np 1 -b 1024 -ub 512 \
    -fit off --no-mmap --mlock \
    --temp 0.3 --top-p 0.95 --top-k 40 \
    --reasoning-budget 8192 \
    --chat-template-kwargs '{"preserve_thinking":true}' \
    --host 0.0.0.0 --port 8080
```

### Результати після оптимізації

| Метрика | До | Після | Зміна |
|---|---|---|---|
| Generation throughput | 47.20 ток/с | **~55-58 ток/с** | **+17-23%** |
| Draft acceptance rate | ~70% | **71-88%** | вища якість драфта |
| VRAM | 9 970 / 11 264 MiB | **10 572 / 11 264 MiB (93.8%)** | щільніше, але стабільно |
| Reasoning budget | 2048 | **8192** | глибший аналіз коду |
| Температура | 0.15 | **0.3** | краща креативність |

### Ключові інсайти

1. **Draft KV cache quantization** — вирішальний параметр для MTP на 11GB карті. Без `-ctkd q8_0 -ctvd q8_0` MTP + 64K контекст неможливі через OOM при завантаженні.
2. **n_max=2 краще ніж 3** — при acceptance length ~2.09, генерація третього draft-токена додає оверхед без користі.
3. **`-ngl 999` vs `-ngl N`** — з `-ngl 999` система сама оптимізує розподіл шарів між GPU/CPU. Фіксоване `-ngl 26` дало лише 26.5 t/s.
4. **Reasoning budget 8192** — для складного рефакторингу та code review модель потребує >2048 токенів на роздуми. 8192 — практичний максимум для 64K контексту.
