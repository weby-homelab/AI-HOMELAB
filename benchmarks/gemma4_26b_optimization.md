# ⚡ Оптимізація та Бенчмарк Gemma 4 26B MoE MTP на Домашній Робочій Станції

Цей звіт описує досвід запуску, оптимізації та бенчмаркінгу великої моделі нового покоління **Gemma 4 26B** (архітектура Mixture of Experts - MoE) у локальних умовах домашньої лабораторії з обмеженими апаратними ресурсами.

---

## 🖥️ Апаратна конфігурація (Hardware Spec)
Тестування проводилося на типовому бюджетному хості для локального штучного інтелекту:
* **Процесор:** Intel Xeon E5-2666 v3 (10 фізичних ядер / 20 потоків, Haswell)
* **Оперативна пам'ять:** 128 GB DDR4 RAM
* **Відеокарта:** NVIDIA GeForce RTX 2080 Ti 11GB VRAM (Blower)
* **Накопичувач:** NVMe M.2 SSD
* **Операційна система:** Linux (Ubuntu/WSL)

---

## 📦 Моделі та Інференс-платформа
* **Основна модель:** `Gemma4-26B-A4B-QAT-Uncensored-HauhauCS-Balanced-Q4_K_M.gguf` (~16.8 ГБ)
* **Draft-модель (MTP):** `mtp-gemma-4-26B-A4B-it.gguf` (~252 МБ)
* **Рушій:** `llama-server` (llama.cpp версії `9826` із підтримкою CUDA 13.3)
* **Тип розподілу (GPU Offload):** Гібридний інференс (14 шарів моделі вивантажено на GPU, 34 шари та KV-Cache прораховуються на CPU).

---

## 🛠️ Оптимізаційні кроки (Performance Tuning)

Для досягнення максимальної швидкості та стабільності системи на великих контекстах було виконано комплексне налаштування ОС та сервера:

1. **Спекулятивне декодування (MTP):**
   * Використано MTP-драфтер з параметрами `--spec-type draft-mtp --spec-draft-n-max 2` для паралельного передбачення токенів.
   * Вилучено прапорець `--embedding` для усунення конфліктів у побудові графів MTP.

2. **Блокування пам'яті (Memory Pinning):**
   * Активовано прапорець `--mlock` у `llama-server` для утримання ваг моделі у фізичній ОЗП та запобігання виселення у swap.
   * Додано ліміти `LimitMEMLOCK=infinity` та `LimitNOFILE=1048576` у systemd-юніт сервісу.

3. **Оптимізація ОС:**
   * Переведено усі ядра CPU у режим максимальної частоти `performance` (через cpu-governor).
   * Встановлено `vm.swappiness=10` для зниження агресивності скидання даних у swap.

4. **Квантування KV-Cache (Головне виправлення для довгих промптів):**
   * Додано параметри `-ctk q8_0` та `-ctv q8_0` для квантування Key та Value кешу до 8-біт. Це скоротило обсяг оперативної пам'яті під KV-Cache вдвічі, розвантажило шину пам'яті CPU (Memory Bandwidth) і запобігло таймаутам сокетів при великих контекстах.

## ⚡ Результати бенчмарків (Benchmarks)

### 1. Короткі промпти (до 150 токенів, логічні та кодингові задачі)
* **Базовий гібридний інференс (без MTP):** **24.84 t/s** (час відповіді ~5.75 с)
* **З MTP + CPU performance + mlock:** **35.09 t/s** (час відповіді **4.27 c**)
* **Коефіцієнт прийняття токенів драфтером:** **86.2%**
* **Приріст швидкості:** **🚀 +41.2%**

### 2. Великі контексти (46.5K токенів, DevOps лог-аудит)
Тестувався реальний сценарій аналізу 1500 рядків логів та Kubernetes конфігів на адекватність пошуку помилок.
* **Швидкість обробки промпту (Prefill):** **486.48 t/s** (час завантаження 46,571 токенів — **95.73 с**)
* **Швидкість генерації (Inference):** **7.77 t/s** (без зависань чи дискового свопінгу)
* **Точність пошуку (Needle in a Haystack):** **100%** (модель безпомилково знайшла Go Panic та вказала файл, рядок і мітку часу).

## 💡 Ключові висновки (Lessons Learned)

1. **Квантування KV-Cache — мастхев для довгих контекстів:** При CPU-split інференсі Key/Value кеш є основним джерелом завантаження шини пам'яті CPU на довгих промптах. Квантування до `q8_0` (або `q4_0`) розриває це "пляшкове горло" і збільшує швидкість роботи з пам'яттю майже вдвічі, рятуючи від socket timeout.
2. **Спекулятивне декодування на домашньому залізі працює:** Невеликий MTP-драфтер (~250MB) дає колосальний приріст швидкості (+41%) за мінімальних накладних витрат VRAM.
3. **Обмеження VRAM 11GB:** Для моделей розміру 25B+ (таких як Gemma 4) на картках типу RTX 2080 Ti / 3060 гібридний інференс є єдиним виходом. Проте за належної оптимізації ОС та KV-Cache він забезпечує цілком робочу швидкість для повсякденних агентських завдань.
