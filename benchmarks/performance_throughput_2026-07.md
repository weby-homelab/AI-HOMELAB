## 📊 1. llama-bench — Низькорівневе C++ ядро (без HTTP)

Вимірює чисту продуктивність ядра інференсу без накладних витрат HTTP-сервера та мережі.

### 1.1 CPU vs GPU порівняння

| Конфігурація | pp512 @ d2048 (t/s) | tg128 @ d2048 (t/s) | Прискорення |
|-------------|-------------------|-------------------|------------|
| **CPU-only** (ngl=0, ncmoe=26) | 262.41 ± 1.99 | 13.69 ± 0.03 | 1x (база) |
| **GPU-offloaded** (ngl=999, ncmoe=26) | 396.85 ± 2.58 | 49.29 ± 0.54 | **3.6x** |
| **GPU-offloaded** (ngl=999, ncmoe=50) | 296.73 ± 2.63 | 42.57 ± 1.04 | 3.1x |

> **Висновок:** GPU-прискорення дає **3.6-кратний** приріст швидкості генерації. Збільшення `ncmoe` (більше експертів на CPU) знижує продуктивність на ~14%.

### 1.2 Вплив розміру промпту (GPU, ngl=999, ncmoe=26, d=4096)

| Prompt | pp (t/s) | tg128 (t/s) |
|--------|---------|------------|
| pp=128 | 136.49 ± 4.55 | 47.23 ± 0.57 |
| pp=512 | 391.57 ± 3.61 | 47.23 ± 0.57 |
| pp=1024 | 388.94 ± 0.05 | 47.23 ± 0.57 |
| pp=2048 | 392.02 ± 1.22 | 47.23 ± 0.57 |

### 1.3 Вплив довжини генерації (GPU, ngl=999, ncmoe=26, d=4096)

| Gen Tokens | pp512 (t/s) | tg (t/s) |
|------------|------------|---------|
| tg=128 | 393.36 ± 1.69 | 53.35 ± 0.06 |
| tg=256 | 393.36 ± 1.69 | 53.43 ± 0.02 |
| tg=512 | 393.36 ± 1.69 | 52.54 ± 0.29 |

> **Висновок:** Швидкість генерації стабільна незалежно від довжини (52.5-53.4 t/s).

### 1.4 Вплив глибини контексту (GPU, ngl=999, ncmoe=26)

| Depth | pp512 (t/s) | tg128 (t/s) |
|-------|------------|------------|
| d=4096 (4K) | 394.09 ± 1.79 | 53.25 ± 0.07 |
| d=8192 (8K) | 380.55 ± 4.13 | 52.54 ± 0.22 |
| d=16384 (16K) | 376.53 ± 0.83 | 51.57 ± 0.07 |

> **Висновок:** Зі збільшенням контексту від 4K до 16K швидкість prefill падає на ~4.5%, генерація на ~3.2%.

## 📊 2. llama-benchy — Реальний клієнтський досвід (через API з MTP)

Вимірює продуктивність через OpenAI-сумісний API з увімкненим MTP speculative decoding:
- **pp (prompt processing):** t/s обробки вхідного промпту
- **tg (text generation):** t/s генерації відповіді
- **TTFT (Time to First Token):** затримка до першого токена
- **ttfr (Time to First Response):** час до отримання першого токена клієнтом

### 2.1 Базові показники (pp=512, tg=128, depth=2048, c=1)

| Метрика | Значення |
|---------|---------|
| Prompt processing | **364.90 ± 6.16 t/s** |
| Text generation (з MTP) | **60.29 ± 7.28 t/s** |
| TTFT (Time to First Token) | **5,922 ± 115 ms** |
| TTFR (клієнтський) | **5,921 ± 115 ms** |

### 2.2 Швидкість промпту на великому контексті (depth=8192)

| Prompt | pp (t/s) | tg128 (t/s) | TTFT (ms) |
|--------|---------|------------|-----------|
| pp=128 | 435.41 ± 3.42 | 56.97 ± 4.40 | 17,296 ± 19 |
| pp=512 | 438.48 ± 10.37 | 61.99 ± 1.18 | 18,118 ± 207 |
| pp=1024 | 439.72 ± 4.29 | 55.01 ± 1.82 | 18,929 ± 272 |
| pp=2048 | 435.80 ± 5.58 | 61.33 ± 1.04 | 21,181 ± 362 |

> **Висновок:** МTP-декодування забезпечує швидкість генерації **55-62 t/s** навіть при контексті 8K. TTFT зростає з об'ємом вхідного контексту (очікувано, через prefill).

### 2.3 Тест навантаження (Concurrency)

| Concurrency | pp (sys t/s) | tg (sys t/s) | tg (per-req t/s) | TTFT (ms) |
|-------------|-------------|-------------|-----------------|-----------|
| 1 clients | 378.13 ± 16.54 | 60.35 ± 5.04 | 60.35 ± 5.04 | 5,956 ± 132 |
| 2 clients | 338.86 ± 1.02 (sys), 278.76 ± 107.09 (avg) | 57.29 ± 3.19 (sys), 24.48 ± 0.27 (avg) | 24.48 ± 0.27 | 10,270 ± 4,061 |

> **Висновок:** При 2 одночасних клієнтах пропускна здатність системи (system-wide tg) падає лише на ~5% (60.35 → 57.29), але кожен окремий клієнт отримує ~24.5 t/s (у 2.5x повільніше) через чергування.

## 📊 3. SPEED-Bench (NVIDIA) — Спеціалізований фреймворк

SPEED-Bench призначений для стандартизованої оцінки продуктивності сервера. Доступні категорії: `qualitative`, `throughput_1k`, `throughput_2k`, `throughput_8k`, `throughput_16k`, `throughput_32k`.

> **Примітка:** Через обмеження VRAM (11 GB) та тривалий час виконання для контекстів >8K, повний прогон SPEED-Bench не було завершено. Для Ornith-1.0-35B на RTX 2080 Ti середній час одного семплу при 16K контексті становить ~2 хвилини.

Структуру SPEED-Bench інтегровано в інфраструктуру тестування для майбутніх прогонів.

## 📈 MTP Speedup Analysis

Порівняння `llama-bench` (без MTP) vs `llama-benchy` (з MTP) дозволяє оцінити реальне прискорення від спекулятивного декодування:

| Режим | tg128 @ d2048 (t/s) | Прискорення |
|-------|-------------------|------------|
| **Без MTP** (llama-bench, GPU) | 49.29 | 1x (база) |
| **З MTP** (llama-benchy, API) | 60.29 | **+22.3%** |
| **З MTP** (llama-benchy, 2 clients) | 57.29 (sys) | **+16.2%** |

> **Висновок:** MTP speculative decoding забезпечує **~22% прискорення** генерації тексту. Це особливо важливо для MoE-моделей, де велика частина шарів вивантажена на CPU.

**Команди для відтворення:**

```bash
# llama-bench (потребує зупинки сервера)
systemctl stop llama-server
/root/beellama.cpp/build/bin/llama-bench -m /mnt/nvme-models/ornith-1.0-35b-MTP-graft-Q4_K_M.gguf -p 512 -n 128 -d 2048 -ngl 999 -ncmoe 26 -t 10 -r 3

llama-benchy --base-url http://127.0.0.1:8080 --model local --served-model-name local --pp 512 --tg 128 --depth 2048 --runs 3

# SPEED-Bench
python3 /root/beellama.cpp/tools/server/bench/speed-bench/speed_bench.py \
  --url http://127.0.0.1:8080 --bench throughput_8k --limit 3
```
