## 🌐 1. Геополітична перевірка та суверенітет (Geopolitical Vetting)

### Чому це критично?
У червні 2026 року ландшафт ШІ став невід'ємною частиною геополітичного протистояння та національної безпеки. Використання моделей, створених компаніями чи дослідницькими групами під юрисдикцією тоталітарних або ворожих держав, несе прямі загрози національному суверенітету, кібербезпеці та цілісності даних. Моделі з високим рівнем геополітичного ризику можуть містити приховані бекдори (backdoors), механізми витоку даних через особливості токенізації, бути навченими на цензурованих або маніпулятивних наборах даних, або ж мати вбудований ідеологічний ухил.

### Категорично заборонені сутності (High-Risk Regions Filter)
Ми впроваджуємо **жорстку заборону** (strict filtering out) на використання моделей та продуктів від компаній та організацій з Російської Федерації (РФ) та Китайської Народної Республіки (КНР).

До чорного списку входять:
*   **DeepSeek** (всі моделі, включаючи DeepSeek-V3, DeepSeek-Coder, DeepSeek-R1) — КНР.
*   **Qwen** (Alibaba Group, включаючи Qwen2.5, Qwen2.5-Coder, Qwen-VL) — КНР.
*   **Yandex / YandexGPT / YaLM** (Яндекс) — РФ.
*   **GigaChat** (Сбербанк) — РФ.

### Ризики та загрози:
1.  **Законодавчий примус до співпраці:** Відповідно до Закону КНР про національну розвідку (2017 р.) та законів РФ про зв'язок (пакет Ярової тощо), будь-яка компанія в цих країнах зобов'язана надавати спецслужбам повний доступ до своєї інфраструктури, вихідного коду, моделей та даних користувачів.
2.  **Забруднення ваг (Weight Poisoning):** Навмисне внесення тригерів під час фази навчання. Модель може працювати нормально, але при виявленні специфічного промпту (напр. певного кодового слова) активувати прихований режим, що дозволяє виконувати шкідливі команди або викривляти результати.
3.  **Ідеологічний та цензурний ухил:** Моделі з КНР жорстко цензуруються відповідно до директив Комуністичної партії Китаю (CAC regulations), що обмежує їхні можливості в об'єктивному аналізі геополітичних подій, історії та стандартів безпеки.

### Довірені альтернативи (Sovereign Alternatives)
Ми рекомендуємо використовувати моделі від розробників, які діють у межах правового поля демократичних країн із захистом авторських прав та конфіденційності:
*   **Meta AI:** Серія моделей LLaMA (LLaMA 3, LLaMA 3.1, LLaMA 3.2).
*   **Google:** Серія моделей Gemma (Gemma 2).
*   **Mistral AI:** Французькі моделі з відкритим кодом (Mistral-7B, Mixtral 8x7B/8x22B, Codestral).
*   **Microsoft:** Моделі сімейства Phi (Phi-3, Phi-3.5).
*   **Apple:** Моделі OpenELM.

## 🔒 2. Приватність даних та ізоляція інференсу (Data Privacy & Isolation)

### Локальний інференс як стандарт (Local-First Default)
Для забезпечення максимальної конфіденційності, обробка промптів має відбуватися **виключно локально** на власній апаратній інфраструктурі (Self-Hosted Inference). Це нівелює загрозу перехоплення даних (Data-in-Transit) або їх аналізу на сторонніх серверах хмарних провайдерів.

Рекомендовані локальні рушії:
*   **Ollama:** Для швидкого та безпечного розгортання моделей.
*   **llama.cpp:** Для низькорівневої оптимізації та мінімального споживання пам'яті.
*   **vLLM:** Для високопродуктивного інференсу з підтримкою багатьох GPU.

### Фаєрвол та обмеження вихідного трафіку (Outbound Egress Filtering)
Навіть локально запущена модель може спробувати здійснити вихідне з'єднання (наприклад, для надсилання телеметрії або через вразливість у виконуваному коді агента).
*   **Zero-Trust мережа:** Сервери інференсу моделей повинні бути розміщені в ізольованому VLAN (наприклад, `VLAN 10: AI Lab`), який не має прямого доступу до інтернету.
*   **Правила nftables:** Весь вихідний трафік від серверів інференсу має бути заблокований, за винятком локальних IP-адрес для оновлення системного ПЗ та авторизованих джерел моделей (Hugging Face / Ollama Registry) за потреби оновлення.

### Захист від телеметрії (Telemetry Protection)
Багато бібліотек машинного навчання за замовчуванням збирають телеметрію та статистику використання. Перед запуском необхідно явно вимикати ці функції:
```bash
# Вимкнення телеметрії для Hugging Face та PyTorch
export HF_HUB_DISABLE_TELEMETRY=1
export DO_NOT_TRACK=1

# Для Ollama (використання локального режиму без автоматичного репортингу)
export OLLAMA_NOPRUNE=1
```

## 🔑 3. Цілісність коду та ваг (Code & Weight Integrity)

### Вразливість формату Pickle (Legacy PyTorch `.ckpt`, `.pt`, `.bin`)
Традиційні моделі PyTorch часто поширювалися у форматі серіалізації Python `pickle`. Цей формат є **критично небезпечним**, оскільки процес десеріалізації дозволяє виконувати довільний Python-код. Завантаження моделі у форматі `.bin` або `.pt` з ненадійного джерела еквівалентне запуску вірусу з правами адміністратора.

> [!CAUTION]
> Завантаження файлів `.bin`, `.pt`, `.ckpt`, `.pth` через стандартний `torch.load()` може призвести до повного злому сервера. **НІКОЛИ** не використовуйте ці формати без попереднього аналізу в пісочниці.

### Безпечні формати моделей
Ми вимагаємо використання виключно безпечних форматів збереження ваг моделей, які не дозволяють виконувати довільний код:
1.  **GGUF (GPT-Generated Unified Format):** Спеціальний бінарний формат від спільноти `llama.cpp`. Містить лише метадані та тензори, є повністю безпечним для завантаження, оптимізований для CPU+GPU інференсу та підтримує швидке відображення пам'яті (mmap).
2.  **Safetensors:** Формат, розроблений Hugging Face. Використовує плоский бінарний макет без підтримки серіалізації коду (no executable code execution).

### Криптографічна верифікація (SHA256 Checksums)
Кожен файл моделі повинен бути верифікований за допомогою SHA256 контрольних сум перед завантаженням у продуктивне середовище.
```bash
# Приклад перевірки цілісності завантаженого файлу моделі
sha256sum llama-3-8b-instruct.Q4_K_M.gguf
# Порівняйте вихідний хеш із офіційним значенням на Hugging Face (розділ LFS Metadata)
```

## ⚡ 4. Продуктивність та енергоефективність (Performance & Efficiency)

### Метрика Tokens per Second per Watt (T/s/W)
В умовах локального хостингу та можливих перебоїв із живленням (блекаути, обмеження потужності домашньої мережі), ефективність використання енергії є критичним параметром. Ми вимірюємо ефективність моделей за метрикою:
$$\text{Efficiency} = \frac{\text{Tokens/sec}}{\text{System Wattage}}$$

Для оптимізації енергоспоживання необхідно:
*   Використовувати **квантування (Quantization)**: Моделі у форматі Q4_K_M або Q8_0 зменшують навантаження на пропускну здатність пам'яті, знижують енергоспоживання GPU/CPU до 40% при втраті точності менше ніж 1-2%.
*   Застосовувати **mmap (Memory Mapping)**: GGUF дозволяє завантажувати модель за допомогою системного виклику `mmap()`, що дозволяє ОС керувати кешем сторінок пам'яті без дублювання даних в оперативній пам'яті.

### Ліміти роботи на резервному живленні (EcoFlow/Battery Limits)
При переході системи на живлення від зарядних станцій (EcoFlow, Bluetti) або інверторів із акумуляторами:
1.  **Автоматичний режим енергозбереження (Blackout Mode):** Оркестратор повинен перемикати запити користувачів на менш ресурсомісткі моделі (наприклад, перехід з LLaMA 3 70B на LLaMA 3.2 3B або Phi-3.5 Mini).
2.  **Апаратні ліміти (Power Limits):** Обмеження максимального споживання GPU через драйвер Nvidia:
    ```bash
    # Обмеження енергоспоживання відеокарти до 150 Вт (на прикладі RTX 3090/4090)
    nvidia-smi -pl 150
    ```

## ⚖️ 5. Ліцензування та комплаєнс (Licensing & Compliance)

### Дозволені ліцензії (Permissive Open-Weights)
Для запобігання юридичним ризикам та забезпечення свободи використання (як для приватних цілей, так і для можливої комерційної інтеграції), моделі повинні поставлятися під дозволеними ліцензіями:

*   **Apache 2.0:** Дозволяє комерційне використання, модифікацію, розповсюдження та субліцензування (напр. Mistral, Phi-3).
*   **MIT:** Максимально вільна ліцензія (напр. деякі моделі від дослідницьких інститутів).
*   **LLaMA 3 License / Gemma Terms of Use:** Спеціальні ліцензії на відкриті ваги (Open-Weights), які дозволяють комерційне використання (з обмеженням на кількість активних користувачів на місяць — наприклад, до 700 млн користувачів для LLaMA 3), але забороняють використання виходів моделі для навчання інших конкурентних моделей.

## ✅ Чеклист верифікації моделі (Vetting Checklist)

Використовуйте цей чеклист перед завантаженням та запуском будь-якої моделі в локальній системі:

- [ ] **Геополітика:** Модель не має зв'язків з розробниками чи компаніями з РФ та КНР (включаючи суб-бренди та форки).
- [ ] **Формат:** Модель завантажена у безпечному форматі: `.gguf` або `.safetensors`. Файли `.bin` або `.pt` відсутні.
- [ ] **Цілісність:** SHA256 хеш завантаженого файлу перевірено та звірено з оригінальним репозиторієм на Hugging Face.
- [ ] **Конфіденційність:** Мережевий ізольований VLAN налаштовано для блокування будь-вого вихідного трафіку від процесу моделі.
- [ ] **Телеметрія:** Змінні середовища `HF_HUB_DISABLE_TELEMETRY=1` та `DO_NOT_TRACK=1` встановлено.
- [ ] **Ефективність:** Модель квантована (Q4_K_M або Q8_0) для роботи в умовах блекаутів, встановлено апаратні ліміти потужності GPU/CPU.
- [ ] **Юридична чистота:** Ліцензія моделі перевірена, відповідає дозволеному використанню (Apache 2.0, MIT або LLaMA 3/Gemma Open-Weights License).
