## 🚀 Вступ та Філософія безпеки

У 2026 році розвиток агентного ШІ (Agentic AI) повністю змінив парадигму безпеки. Раніше основна увага приділялася захисту статичних моделей інференсу. Сьогодні ШІ-агенти динамічно генерують і виконують код (Python REPL, Bash), користуються браузерами для вебрайтінгу та викликають сторонні API-інструменти.

Стандартний контейнерний рушій Docker (`runc`) використовує спільне ядро хоста Linux. Це створює критичні ризики. Для забезпечення стійкості ми маємо використовувати концепцію **багаторівневої оборони (Defense-in-Depth)**:

```
Container (Packaging)
   ↓
Sandbox / System Call Interception (gVisor)
   ↓
MicroVM / Hardware Virtualization (Firecracker)
   ↓
Confidential Computing (TEE: AMD SEV-SNP / Intel TDX)
   ↓
Zero-Trust Networking (Egress filtering via vsock / proxies)
```

## 🎯 Оновлена модель загроз (Threat Model 2026)

Сучасні загрози для ШІ-систем поділяються на три основні класи:

### 1. Виконання довільного коду та втеча з контейнера (Arbitrary Code Execution & Container Escape)
*   **Вектор атаки:** Prompt Injection → Агент отримує шкідливий промпт → Генерація шкідливого коду → Виконання коду в оточенні (напр., Python REPL) → Експлуатація вразливостей ядра Linux або рантайму контейнера (`containerd` / `runc`) для виходу на хост-систему (Container Escape).
*   **Наслідки:** Повний контроль зловмисника над фізичним чи віртуальним сервером хоста.

### 2. Зловживання інструментами та ескалація повноважень (Tool Abuse & Capability Escalation)
*   **Вектор атаки:** Prompt Injection → Агент отримує нову інструкцію (прихований системний промпт) → Агент використовує дозволені інструменти (File Manager, Database Connector, HTTP Client) для несанкціонованого читання приватних файлів, конфігурацій, токенів та секретів, не порушуючи ізоляцію ядра.
*   **Приклад:** Агент зчитує файл `.env` за допомогою законного інструменту читання файлів та відсилає його вміст через дозволений мережевий інструмент на сервер зловмисника.

### 3. Крадіжка моделей та інтелектуальної власності (Model & IP Theft)
*   **Вектор атаки:** Доступ зловмисника до shared-інфраструктури, де хоститься модель, або перехоплення трафіку між вузлами.
*   **Об'єкти викрадення:**
    *   Ваги локальних моделей та LoRA-адаптери.
    *   Бази векторних ембеддінгів (Knowledge Base) та RAG-дані.
    *   Системні інструкції та складні системні промпти (Prompt IP).

## ⚙️ Архітектура ізоляції рантайму (Firecracker vs gVisor vs WASM)

Ці технології є не конкурентами, а різними рівнями та інструментами ізоляції, кожен з яких підходить під свій тип навантаження.

### 1. Firecracker (MicroVMs)
*   **Механізм:** Використовує KVM (Kernel-based Virtual Machine) для створення легковажних віртуальних машин (MicroVMs). Кожне завдання запускається з власним мінімальним ядром Linux.
*   **Переваги:** Максимальна апаратна ізоляція. Час холодного запуску <100 мс. Мінімальне споживання пам'яті (до 5 МБ на MicroVM).
*   **Найкраще для:** Запуску неперевіреного коду, скрапінгу вебсайтів через Headless Chrome, динамічного виконання Python-інструментів.

### 2. gVisor
*   **Механізм:** Перехоплює системні виклики (syscalls) з контейнера за допомогою компонента `Sentry`, що працює у просторі користувача (user-space), та ізолює контейнер від прямої взаємодії з ядром хоста.
*   **Переваги:** Сумісність із OCI/Docker (рантайм `runsc`). Простіша інтеграція у Kubernetes. Нижчий overhead на CPU порівняно з VM.
*   **Найкраще для:** Compute-heavy інференсу моделей, де потрібен захист від escape-вразливостей, але зберігається стандартний Docker-стейк.

### 3. WASM (WebAssembly) & WASI
*   **Механізм:** Виконання коду у строго обмеженій пісочниці байт-коду.
*   **Переваги:** Абсолютно детерміноване та портативне середовище. Майже нульовий overhead.
*   **Найкраще для:** Простих інструментів очищення даних (parsers), валідаторів, плагінів або MCP-інструментів (Model Context Protocol).

### Рекомендації щодо вибору ізоляції (Workload Isolation Table)

| Тип навантаження (Workload) | Рівень ризику | Рекомендована ізоляція | Альтернатива / Додатковий рівень |
| :--- | :--- | :--- | :--- |
| **Simple Read-Only Agent** | Низький | **gVisor** (`runsc` в Docker) | Стандартний Docker з AppArmor/Seccomp |
| **MCP Tools, Parsers, Validators** | Низький | **WASM** / **WASI** | gVisor |
| **Code Execution (Python REPL)** | Високий | **Firecracker MicroVM** | gVisor (з лімітом ресурсів та seccomp) |
| **Browser Agents & Web Scraping** | Високий | **Firecracker MicroVM** | gVisor + ізольована VLAN-мережа |
| **Fine-tuning / Training** | Критичний | **Confidential VM / Bare Metal** | Firecracker з підтримкою vGPU |

## 🔒 Конфіденційні обчислення та TEE

Коли ваші моделі розгорнуті у публічній хмарі (AWS, GCP, Azure), ізоляції процесів ОС недостатньо. Адміністратор хмари (або зловмисник, що отримав доступ до хоста) може перехопити дані прямо з оперативної пам'яті.

Для захисту **Data-in-Use** застосовуються технології **Trusted Execution Environments (TEE)**:

### 1. Апаратне шифрування RAM (AMD SEV-SNP / Intel TDX)
*   Пам'ять віртуальної машини повністю шифрується на рівні процесора.
*   Ключі шифрування генеруються апаратно всередині процесора і недоступні гіпервізору хост-системи.
*   Унеможливлює зчитування ваг моделей чи приватних RAG-даних через memory dumping хоста.

### 2. AWS Nitro Enclaves
*   Спеціальні ізольовані віртуальні машини, підключені до основної інстанції через віртуальний сокет (`vsock`).
*   Не мають зовнішньої мережі, дисків чи інтерактивного доступу (`ssh`).
*   Ідеально підходять для обробки чутливих API-ключів та проведення конфіденційного інференсу LLM.

### 3. Confidential GPUs (Nvidia H100 / B200 / Blackwell)
*   Захищають канал передачі даних (PCIe link) між CPU та GPU за допомогою шифрування на льоту.
*   Модель розшифровується тільки всередині апаратно захищеної пам'яті графічного процесора (HBM3).

## 🌐 Мережевий Zero-Trust та Egress Filtering

Найпростіший спосіб вкрасти дані чи токени API для ШІ-агента — це надіслати HTTP/DNS-запит на сервер зловмисника. Мережева ізоляція має бути налаштована на повну заборону неавторизованого вихідного трафіку.

### 1. TLS Inspection & Egress Proxying (Envoy / Squid)
*   Всі вихідні мережеві запити від агента перенаправляються через Egress Proxy.
*   Проксі перевіряє домени за білим списком (Allowlist), наприклад: `api.openai.com`, `huggingface.co`.
*   Спроби з'єднання з будь-якими іншими доменами чи прямими IP-адресами автоматично блокуються.

### 2. Запобігання DNS-тунелюванню (DNS Tunneling Prevention)
*   Зловмисники можуть закодувати вкрадені токени у субдомени DNS-запитів (наприклад, `txt.leak-api-key-here.malicious.com`).
*   **Рішення:** Використовувати локальний DNS-резолвер (напр. CoreDNS) з налаштованим лімітом ентропії імен та блокуванням підозрілих вихідних DNS TXT/CNAME запитів.

### 3. Vsock як заміна TCP/IP
*   Для зв'язку оркестратора з ізольованими агентами (наприклад, у Firecracker) рекомендується повністю відмовитися від TCP-портів.
*   Використовуйте протокол `vsock` (AF_VSOCK), який працює через Unix Domain Sockets хоста. Це повністю прибирає мережевий стек всередині пісочниці, ліквідуючи загрозу сканування внутрішніх мереж.

## ✅ Чеклист впровадження ізоляції

Використовуйте цей чеклист під час проектування архітектури безпеки для ваших AI-застосунків:

- [ ] **Рівень виконання:** Для виконання генерованого коду використовується Firecracker або gVisor (замість стандартного Docker runc).
- [ ] **Рівень ресурсів:** Для кожного контейнера/VM жорстко встановлено ліміти CPU, RAM та дискового простору (tmpfs/Read-only root).
- [ ] **Рівень мережі:** Заблоковано весь вихідний трафік, крім білого списку доменів (Egress Allowlist proxy).
- [ ] **Рівень DNS:** DNS-запити проходять через контрольований шлюз із моніторингом тунелювання.
- [ ] **Рівень інструментів:** Агенту заборонено доступ до локальних системних файлів поза його пісочницею. Всі секрети та ключі монтуються як тимчасові змінні середовища, до яких у коду агента немає прямого доступу.
- [ ] **Апаратний рівень (хмара):** Для чутливих даних інференс запускається у Confidential VMs (Intel TDX / AMD SEV-SNP).
