# 🔋 БЛЕКАУТ-ГАЙД: Налаштування ШІ-лабораторії для роботи від EcoFlow

Цей гайд призначений для українських розробників та ентузіастів, які будують локальні AI-лабораторії в умовах нестабільного енергопостачання. Тут описано, як мінімізувати споживання енергії вашим залізом під час роботи від акумуляторів чи зарядних станцій (EcoFlow, Bluetti тощо).

---

## 🔌 1. Апаратна енергоефективність (Hardware Choice)

Вибір заліза визначає, скільки годин пропрацює ваша лаба під час відключення світла.

| Платформа | Робоче споживання | Споживання під навантаженням (LLM) | Оцінка автономності (на EcoFlow River 2, 256Wh) |
|---|---|---|---|
| **Apple Silicon (Mac mini M2/M4)** | 4–7 W | **15–25 W** | **~10-12 годин чату** |
| **Міні-ПК (Ryzen 7 7840HS)** | 8–12 W | **35–55 W** | **~4-5 годин чату** |
| **PC з RTX 3060 12GB (x86)** | 60–80 W | **180–220 W** | **~1 година чату** |
| **PC з RTX 4060 Ti 16GB** | 50–70 W | **140–170 W** | **~1.5 години чату** |

> [!TIP]
> **Apple Silicon** є абсолютним переможцем для блекаутів. Оскільки оперативна пам'ять є уніфікованою, моделі до 32B параметрів виконуються безпосередньо на NPU/GPU з мінімальним виділенням тепла та споживанням енергії.

---

## 🏎️ 2. Оптимізація x86 систем з GPU (Nvidia RTX)

Якщо ви використовуєте класичний ПК з дискретною відеокартою Nvidia, під час роботи від батареї обов'язково оптимізуйте ліміти живлення.

### Обмеження споживання GPU (Power Limit)
Сучасні відеокарти Nvidia мають дуже гнучкі ліміти живлення. Обмеження ліміту на 40-50% знижує продуктивність всього на 10-15%, але колосально економить заряд EcoFlow.

**Linux (потрібні права root):**
```bash
# Перевірити поточні ліміти та споживання
nvidia-smi -q -d POWER

# Встановити максимальний Power Limit (наприклад, 100W для RTX 3060, яка зазвичай споживає 170W)
sudo nvidia-smi -pl 100
```

**Windows:**
Використовуйте утиліту **MSI Afterburner** або офіційний `nvidia-smi` через командний рядок (CMD від імені адміністратора):
```cmd
nvidia-smi -pl 100
```

### Зниження частоти та напруги (Undervolting)
Зниження робочої напруги GPU через криву частот (Curve Editor в MSI Afterburner) дозволяє зекономити ще 20–30W без будь-якої втрати швидкості генерації токенів (t/s).

## 💻 3. Оптимізація систем без GPU (CPU-only)

Якщо моделі запускаються на процесорі (через CPU-інференс Ollama / llama.cpp):

### Обмеження кількості потоків (Threads Limit)
Запуск інференсу на 100% ядер процесора викликає експоненційне зростання споживання енергії (TDP процесора зростає з 15W до 65W+).
Оптимальна кількість потоків для `llama.cpp` дорівнює кількості **фізичних P-ядер** (performance cores) вашого процесора, або половині логічних потоків.

**Конфігурація Ollama через змінну середовища:**
Встановіть кількість потоків (наприклад, 4 потоки для стабільної роботи):
```bash
export OLLAMA_NUM_PARALLEL=1
# Обмеження потоків для внутрішнього двигуна llama.cpp (зазвичай керується динамічно, але можна зафіксувати через конфіг)
```

## 🐳 4. Налаштування Docker та ОС під час блекаутів

### Вимкнення фонових сервісів
Зупиніть усі важкі контейнери, які не потрібні для роботи ШІ-моделей (наприклад, локальні бази даних, моніторинг Grafana/Prometheus):
```bash
docker stop $(docker ps -q | grep -v open-webui | grep -v ollama)
```

### Налаштування Power Governor (Linux)
Переведіть процесор у режим енергозбереження:
```bash
# Для процесорів Intel/AMD
sudo cpupower frequency-set -g powersave
```

## 🔋 5. Рекомендовані сценарії роботи від станцій

1. **Сценарій "Максимум автономності" (до 20+ годин)**:
   * Хост: Mac Mini M2/M4 (8W середнє споживання) або MacBook Air M2/M4.
   * Моделі: **Gemma 4 E2B/E4B** (~2B/4B) або **Phi-4 mini 3.8B**.
   * Квантизація: **`Q3_K_L`** або **`IQ3_XS`** (екстремальна 3-бітна квантизація значно розвантажує шину пам'яті та економить енергію).
   * Оптимізація інференсу: запуск через Ollama з обмеженням `OLLAMA_KEEP_ALIVE=10s` (щоб NPU/GPU миттєво засинав після кожної відповіді) та лімітом довжини відповідей (наприклад, `max_tokens = 150`). Запуск безпосередньо без Docker (Docker споживає додаткові 2-3W).
2. **Сценарій "Компроміс" (до 6 годин)**:
   * Хост: Ноутбук з RTX 4060 Laptop GPU.
   * Модель: **Gemma 4 12B** (`Q4_K_M`) або **Phi-4 Multimodal 5.6B** (`Q4_K_M`).
   * Налаштування заліза: Power Limit GPU встановлено на **40-45W**.
   * Екран ноутбука вимкнено або яскравість на мінімумі, підключення через SSH/браузер з іншого пристрою (наприклад, планшета чи телефона, які споживають менше енергії).

## 📦 6. Екосистема AI Blackout Stack (Sovereign Offline AI)

Для створення повноцінної автономної AI-системи, яка переживе тривалі відключення світла та зв'язку, недостатньо просто обмежити споживання процесора. Потрібно побудувати зв'язну архітектуру: **UPS + Starlink + Tailscale + Local LLM + Offline RAG + Power Budgeting**.

```
                       🔋 ЕКОСИСТЕМА BLACKOUT STACK

                 ┌───────────────────────────────────────┐
                 │       ДБЖ (UPS) з підтримкою NUT      │
                 └──────────────────┬────────────────────┘
                                    │ (USB / SNMP моніторинг)
                                    ▼
                 ┌───────────────────────────────────────┐
                 │     AI-Сервер (Power Budgeting)       │
                 │   - Автоматичний Power Limit GPU      │
                 │   - Зупинка важких Docker-сервісів   │
                 └──────────────────┬────────────────────┘
                                    │
           ┌────────────────────────┴────────────────────────┐
           ▼                                                 ▼
┌──────────────────────┐                          ┌──────────────────────┐
│  Starlink (12V PoE)  │                          │     Local LLM &      │
│  - Bypass Mode       │                          │     Offline RAG      │
│  - Без 220V інвертора│                          │  - Векторна БД на SSD│
└──────────┬───────────┘                          └──────────────────────┘
           │ (Mesh VPN)
           ▼
┌──────────────────────┐
│  Tailscale Remote    │
│  - Доступ за CGNAT   │
└──────────────────────┘
```

### 1. Інтеграція ДБЖ (UPS) та NUT (Network UPS Tools)
Для автоматичного реагування на зміну джерела живлення підключіть ДБЖ до вашого AI-сервера через USB та встановіть службу `nut`:

```bash
sudo apt update && sudo apt install nut -y
```

Налаштуйте `/etc/nut/nut.conf` для роботи в режимі standalone:
```ini
MODE=standalone
```

Створіть скрипт тригера `/etc/nut/upssched.conf` або підключіться до стандартного демона, щоб при переході ДБЖ на акумулятор (`ONBATT`) сервер виконував автоматичне скидання навантаження (load shedding) — наприклад, перемикав ліміт споживання відеокарти до мінімуму або зупиняв сервіси vLLM та залишав лише легку Ollama.

### 2. Оптимізація живлення Starlink (12V/48V PoE Mod)
Стандартний Starlink (Gen 2 або Gen 3) споживає **50–75W**, оскільки працює через мережу 220V і використовує неефективний вбудований інвертор (втрати 15-20% при перетворенні 12V/48V батареї -> 220V AC -> 48V DC для антени).

**Кроки оптимізації:**
1. **Bypass Mode:** Переведіть Starlink у режим Bypass, щоб вимкнути Wi-Fi роутер Starlink. Використовуйте енергоефективний роутер на OpenWrt (наприклад, GL.iNet, споживання 3-5W).
2. **PoE Модифікація:** Використовуйте спеціалізований перетворювач напруги 12V->48V Step-Up та інжектор PoE (наприклад, Dishy-in-PoE адаптер). Це дозволить живити антени безпосередньо від акумулятора 12V/24V постійного струму, знижуючи споживання всієї системи Starlink до **35–45W** (економія до 40% ємності EcoFlow).

### 3. Мережева стійкість через Tailscale Mesh VPN
Під час відключення світла мобільні вежі швидко деградують, а публічні IP-адреси провайдерів зникають або постійно змінюються. Якщо сервер підключено через Starlink або резервний 4G-модем, він працюватиме за CGNAT (Carrier-Grade NAT), що робить звичайний порт-форвардинг неможливим.

**Рішення:**
* Встановіть **Tailscale** на AI-сервер та ваші клієнтські пристрої (ноутбук, телефон).
* Tailscale створює безпечну mesh-мережу на базі протоколу WireGuard. Вона автоматично обходить CGNAT, підтримує зв'язок при зміні IP-адрес та забезпечує доступ до вашого `Open WebUI` чи SSH сервера з будь-якої точки світу без публічної IP-адреси.

### 4. Повністю Offline RAG & Кешування знань
Для повноцінної автономності ваші ШІ-агенти не повинні залежати від сторонніх хмарних інструментів чи онлайн-пошуку.
* **Локальний Embedding:** Налаштуйте n8n або LangGraph на використання виключно локальної моделі `nomic-embed-text` або `all-minilm` замість OpenAI Embeddings.
* **Синхронізація у вікнах зв'язку:** Налаштуйте cron-задачі, які завантажують необхідні документи, статті чи новини у векторну базу Qdrant, коли інтернет-зв'язок стабільний. Під час повної відсутності зв'язку агенти працюватимуть суто з локально кешованими даними.

### 5. Автоматизований скрипт Power Budgeting
Створіть скрипт, який запускається при переході системи на батарею та оптимізує споживання заліза:

```bash
#!/usr/bin/env bash
# /usr/local/bin/ai-power-budget.sh
# Скрипт автоматичної оптимізації енергоспоживання ШІ-лабораторії

set -euo pipefail

# Перевірка джерела живлення (для Linux хостів)
POWER_SOURCE=$(cat /sys/class/power_supply/AC/online 2>/dev/null || echo "1")

if [ "$POWER_SOURCE" = "0" ]; then
    echo "🔋 Робота від батареї! Перехід у режим енергозбереження..."

    # 1. Обмеження відеокарти Nvidia (якщо є)
    if command -v nvidia-smi &> /dev/null; then
        nvidia-smi -pl 80 || true # Встановити ліміт 80W для RTX 3060
    fi

    # 2. Переведення процесора у режим powersave
    if command -v cpupower &> /dev/null; then
        cpupower frequency-set -g powersave || true
    fi

    # 3. Зупинка важких Docker-контейнерів (напр. моніторинг, vLLM)
    # Залишаємо лише базову Ollama та Open WebUI
    docker stop agent-litellm agent-n8n agent-qdrant || true

    # 4. Обмеження потоків Ollama
    export OLLAMA_NUM_PARALLEL=1

else
    echo "🔌 Робота від мережі. Відновлення максимальної продуктивності..."

    # 1. Скидання лімітів GPU
    if command -v nvidia-smi &> /dev/null; then
        nvidia-smi -rac || true # Відновити стандартні ліміти
    fi

    # 2. Процесор у режим performance
    if command -v cpupower &> /dev/null; then
        cpupower frequency-set -g performance || true
    fi

    # 3. Запуск усіх сервісів
    docker start agent-litellm agent-n8n agent-qdrant || true
fi
```
