## 🧠 Що таке квантизація?

**Квантизація (Quantization)** — це процес конвертації числових ваг моделі штучного інтелекту з високоточних форматів з рухомою комою (наприклад, 16-бітних `FP16` або `BF16`) у формати нижчої розрядності (наприклад, 8, 4, 3 або навіть 2-бітні цілі числа `INT`).

### ⚖️ Основні переваги:
* **Зменшення розміру моделі**: 8-мільярдна модель у форматі `FP16` займає близько 16 GB. У форматі квантизації `Q4_K_M` (4 біти) її розмір зменшується до ~4.8 GB.
* **Прискорення інференсу (t/s)**: Зменшений об'єм моделі дозволяє швидше завантажувати ваги в пам'ять. Пропускна здатність пам'яті (Memory Bandwidth) є основним обмежувачем швидкості локальних LLM.
* **Зниження енергоспоживання**: Менше циклів читання з пам'яті = менше виділення тепла та споживання енергії (критично для роботи від EcoFlow).

> [!IMPORTANT]
> Квантизація неминуче призводить до певної втрати точності (перплексії) моделі, проте сучасні методи (наприклад, квантизація типу **K-Quants** або **IQ-Quants**) зводять цю втрату до мінімуму, практично непомітного в реальних завданнях.

## 📊 Порівняння форматів квантизації

Основний контейнер для локальних моделей — формат **GGUF** (GPT-Generated Unified Format), розроблений командою `llama.cpp`. Він містить усі метадані та ваги в одному файлі.

| Метод (GGUF) | Бітність | Втрата якості | Рекомендація |
| :--- | :--- | :--- | :--- |
| **FP16 / BF16** | 16 | ❌ Відсутня | Лише для серверів з надлишком VRAM або для подальшого квантування. |
| **Q8_0** | 8.5 | 🟢 Практично нульова | Найкраща точність, але вимагає багато пам'яті. |
| **Q5_K_M** | 5.5 | 🟢 Вкрай низька | Відмінний баланс, якщо дозволяє VRAM. |
| **Q4_K_M** | 4.8 | 🟢 Дуже низька | 🏆 **Золотий стандарт** для щоденного використання. |
| **Q3_K_L** | 3.5 | 🟡 Помітна | Рекомендовано для блекаутів (максимальна економія енергії). |
| **IQ3_XS** | 3.0 | 🟡 Помітна | Екстремальне стиснення з використанням матриць важливості (Importance Matrix). |
| **Q2_K** | 2.5 | 🔴 Висока | Не рекомендується для загальних завдань через деградацію мовлення. |

## 💾 Розрахунок необхідної пам'яті (VRAM Budgeting)

Перед запуском моделі необхідно розрахувати ліміт пам'яті за формулою:

$$\text{Необхідна VRAM} = \text{Розмір GGUF-файлу} + \text{Розмір KV-кешу} + \text{Оверхед ОС та UI (1.5 - 2 GB)}$$

### Формула розрахунку розміру моделі:
$$\text{Розмір файлу (GB)} \approx \text{Кількість параметрів (B)} \times \frac{\text{Кількість біт на вагу}}{8} \times 1.15 \text{ (метадані)}$$

* Наприклад, для **Phi-4 (14B)** у форматі **Q4_K_M** (~4.8 біт на вагу):
  $$14 \times \frac{4.8}{8} \times 1.15 \approx 9.66 \text{ GB}$$
  З урахуванням контексту (KV Cache) на 8K токенів (~1.5 GB) та ОС, модель комфортно поміститься в карту з **12 GB VRAM** (наприклад, RTX 3060/4060).

## 🛠️ Покрокове квантування власної моделі через `llama.cpp`

Якщо потрібної моделі немає на HuggingFace у форматі GGUF, її можна квантувати самостійно.

### Крок 1: Підготовка оточення
Найпростіший спосіб запустити `llama.cpp` без ручного збирання з вихідного коду — використання Docker.

```bash
# Клонуємо офіційний репозиторій
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
```

### Крок 2: Конвертація моделі HuggingFace (Safetensors) у GGUF
Переконайтеся, що модель завантажена у вихідному форматі (наприклад, `Gemma-4-12B-It` у форматі Safetensors) у папку `./models/my_model`.

Встановлюємо Python-залежності для конвертера:
```bash
pip install -r requirements.txt
```

Виконуємо конвертацію в базовий неквантований формат `FP16 GGUF`:
```bash
python convert_hf_to_gguf.py ./models/my_model --outfile ./models/my_model_fp16.gguf
```

### Крок 3: Виконання квантизації
Запускаємо вбудований квантизатор `llama-quantize` (рекомендується використовувати Docker-образ для підтримки CUDA acceleration):

```bash
docker run -v $(pwd)/models:/models ghcr.io/ggerganov/llama.cpp:light-cuda \
  /app/llama-quantize /models/my_model_fp16.gguf /models/my_model_q4_k_m.gguf Q4_K_M
```

Після виконання у папці `models` з'явиться готовий файл `my_model_q4_k_m.gguf`, готовий до використання.

## 🔌 Запуск кастомної GGUF-моделі в Ollama через `Modelfile`

Щоб інтегрувати отриманий файл у локальний сервіс Ollama:

### 1. Створіть конфігураційний файл `Modelfile`
У тій самій папці, де лежить файл `.gguf`, створіть файл з назвою `Modelfile`:

```dockerfile
# Вказуємо шлях до локального файлу моделі
FROM ./my_model_q4_k_m.gguf

# Встановлюємо параметри контексту та температури
PARAMETER num_ctx 8192
PARAMETER temperature 0.7
PARAMETER stop <|im_end|>

# Задаємо системний промпт для моделі
SYSTEM """
Ти — професійний та лаконічний ШІ-помічник. Відповідай українською мовою.
Дотримуйся вимог безпеки даних та не згадуй сторонні хмарні сервіси.
"""
```

### 2. Створіть модель в Ollama
Виконайте команду у терміналі:

```bash
ollama create my-custom-model -f ./Modelfile
```

### 3. Перевірте роботу
Запустіть модель у чаті:

```bash
ollama run my-custom-model
```
Тепер ваша модель доступна також і через API Ollama (на порті `11434`), і її можна вибрати в інтерфейсі **Open WebUI**.

## 🚀 Рекомендовані конфігурації для 2026 року

Для оптимальної продуктивності на типовому обладнанні домашніх лабораторій використовуйте такі налаштування:

* **Для GPU з 8 GB VRAM (RTX 4060, Apple Silicon 8GB)**:
  * Модель: `Gemma-4-E4B` у кванті `Q8_0` (висока точність) або `Phi-4-mini` у кванті `Q5_K_M`.
* **Для GPU з 12 GB VRAM (RTX 3060/4070, Apple Silicon 16GB)**:
  * Модель: `Phi-4-14B` у кванті `Q4_K_M` або `Gemma-4-12B` у кванті `Q5_K_M`.
* **Для GPU з 16 GB+ VRAM (RTX 4060 Ti 16GB, Apple Silicon 24GB+)**:
  * Модель: `Gemma-4-26B` (MoE) у кванті `Q4_K_M` (вимагає ~18-19 GB загальної пам'яті разом із контекстом).
